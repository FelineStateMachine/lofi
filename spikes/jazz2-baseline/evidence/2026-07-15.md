
Browser procedure (using `playwright-cli`; equivalent DevTools actions are acceptable):

1. Start `deno task spike:jazz2:dev --port 4177 --strictPort`, open the printed URL, and snapshot.
2. Add `retained-alpha53-control`; wait until the UI says `confirmed local`; reload and verify the
   row remains.
3. Get the row ID from `li[data-note-id]` and call
   `globalThis.__LOFI_JAZZ2_PROBE__.update(id, "after-update")`; verify the subscribed list and
   durability state change, then reload again.
4. Open a second tab at the same origin, add `multitab-follower-write`, and verify it appears in
   both tabs.
5. Set the browser context offline, add `offline-local-write`, wait for local confirmation, restore
   the network, and verify both tabs converge without reloading.
6. Start `deno task spike:jazz2:cloud --port 4178 --strictPort`, add `cloud-global-alpha53`, and
   wait until the UI says `global write confirmed`.
7. Save browser storage state to ignored `.playwright-cli/cloud-identity-state.json`, load it in a
   fresh browser context, open the cloud URL, and verify the cloud row downloads. The state file
   contains the bearer identity and must never be committed or printed.

The cloud task publishes the spike schema to the configured Jazz application. It projected only the
public app ID and server URL to the client. The test retained one note in that application; no
credential value is recorded here.

## Compatibility findings

1. Deno's npm layout puts `jazz-wasm` under `node_modules/.deno`. Jazz resolves the WASM to an
   absolute path, while Vite initially allows only the nested spike root. The browser received HTTP
   403 until the Vite configuration allowed the repository root. This fix is required for the nested
   Astro prototype too.
2. With the raw vendor command on a clean first local run, Jazz writes `VITE_JAZZ_APP_ID` into the
   spike's ignored `.env`. Vite restarts while starting and Deno/Vite 8 failed with
   `Listen method has been called more than
   once without closing.` A rerun started in 102 ms. The
   lofi task runner now pre-seeds a stable internal app ID; a clean wrapped first run started in 103
   ms without a restart. The raw failure is retained because the integrated prototype must keep this
   edge.
3. A Vite configuration restart can change the generated local server URL while a persistent browser
   broker is alive. Jazz then raised `IncompatibleBrowserBrokerConfigurationError` and required a
   page reload. The prototype must keep the runtime singleton stable and make reload recovery
   explicit.
4. The plugin's TTY-only local server banner prints its generated admin secret. lofi cannot expose
   that banner unchanged under DX-ENV-01.
5. `BrowserAuthSecretStore` uses localStorage and documents a non-atomic check-then-write on the
   first concurrent visit. First-run multi-tab identity creation needs lofi coordination or a
   documented limitation.
6. The browser console emitted Jazz WASM `CatalogueWriteDenied` warnings while both local and global
   application-row waits succeeded. This is non-blocking for the baseline but must stay visible
   during #5/#6; lofi must not suppress it without proving that it is harmless catalogue traffic.

## Runtime topology and transport

The exact tarball's persistent-browser path uses a SharedWorker broker to elect one leader tab. The
leader owns the module worker that runs the persistent WASM/OPFS path; follower tabs attach through
message ports. The upstream application-data transport is a Rust-owned WebSocket opened by the
runtime. The package's `sync-transport.js` explicitly says the earlier HTTP/SSE path was removed.
Schema publication by the development plugin remains an HTTP control-plane operation and is not the
application-data sync transport.

## Gate decision

**Go** for the Preact and Astro/Deno integration spikes using this pin and the lofi-owned runtime
edge. The edge already prevents the clean-first-run restart and preserves the Vite filesystem
allowance. Do not graduate M1 until the integrated runtime owns singleton/HMR lifecycle and the
public command avoids the vendor secret-printing banner. #4 and #8 remain independent gates for
physical-device durability and recovery semantics.
